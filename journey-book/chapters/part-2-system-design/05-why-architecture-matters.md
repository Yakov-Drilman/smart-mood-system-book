# פרק 5 – למה צריך ארכיטקטורה

עכשיו, כבר לא היה לי ספק מה אני בונה.  
לא אפליקציה קטנה. לא דמו. לא “פרויקט צד”. אלא מערכת.  
וכשאתה מבין שאתה בונה מערכת מגיע רגע שאתה לא יכול לברוח ממנו יותר.  
השאלה “איך לכתוב קוד”? הופכת לשאלה “איך למנוע מהקוד להתפרק בעוד חודשיים”?

---

## השינוי השקט שקורה לכל מפתח

בהתחלה אתה חושב על פיצ׳רים,  
להוסיף טופס, לשמור נתון, להציג גרף.  
אבל בשלב מסוים אתה מתחיל לחשוב אחרת,  
מה יקרה כשיהיו 10,000 משתמשים?  
מה יקרה כשנוסיף AI?  
מה יקרה כשנרצה לשנות את מבנה הרגש?

ופה בדיוק נכנסת הארכיטקטורה.

---

## ארכיטקטורה היא לא טכנולוגיה

זו טעות נפוצה לחשוב שארכיטקטורה זה:  
Node.js  
React  
MongoDB  
AWS

אבל בפועל, ארכיטקטורה היא הדרך שבה חלקי המערכת מדברים אחד עם השני בלי להסתבך.

---

## הבעיה שלי בלי ארכיטקטורה

בהתחלה, בלי לשים לב, המערכת הייתה מתחילה להיראות ככה,  
React קורא API.  
API עושה גם לוגיקה.  
לוגיקה מתערבבת עם שמירה.  
שמירה מושפעת מ-UI decisions.

זה נראה “בסדר” בהתחלה. עד שזה נשבר.

---

## סימן ראשון שמשהו לא נכון

הסימן הראשון היה פשוט,  
כל שינוי קטן שבר משהו אחר.

שינית מבנה רגש → נשבר API.  
שינית API → נשבר UI.  
שינית UI → נשבר data flow.

זה הרגע שבו אתה מבין, אין בעיה בקוד, יש בעיה במבנה.

---

## ההחלטה, לבנות גבולות ברורים

בשלב הזה הגדרתי כלל אחד,  
כל חלק במערכת חייב לדעת רק מה שהוא צריך לדעת ולא יותר.

זה התחיל להיראות כמו חלוקה אמיתית,

UI Layer (React) רק מציג ומעביר פעולות.  
API Layer (Node.js) רק מקבל, מאמת, ומעביר הלאה.  
Service Layer כאן נמצאת הלוגיקה האמיתית.  
Data Layer רק שמירה ושליפה.

---

## הארכיטקטורה החדשה (בפועל)

```text
[ React UI ]
     ↓
[ API Routes ]
     ↓
[ Service Layer ]
     ↓
[ Data Access Layer ]
     ↓
[ Database / JSON ]
```

וזה שינוי קריטי.  
כי פתאום, כל חלק יודע מה התפקיד שלו, אין “קפיצות” בין שכבות, אין תלות מיותרת

---

## ההבנה החשובה ביותר

ארכיטקטורה לא נועדה לעשות את החיים קלים יותר בהתחלה.  
היא נועדה לעשות דבר אחד, למנוע מהמערכת להתמוטט כשהיא גדלה.

---

## הפיתוי המסוכן

היה רגע שבו שקלתי לוותר על זה.  
כי האמת פשוטה, יותר מהר לבנות בלי שכבות.

אבל יש מחיר נסתר,  
קוד מהיר = חוב טכני  
חוב טכני = כאב עתידי  
כאב עתידי = מערכת שלא שורדת

---

## ההחלטה ששינתה את כל הפרויקט

בשלב הזה קיבלתי החלטה סופית,  
אני לא בונה “פיצ׳רים”, אני בונה מערכת שאפשר להרחיב בלי פחד.

וזה אומר,  
לא מערבבים לוגיקה עם UI.  
לא מערבבים DB עם API.  
לא “מאלתרים פתרונות זמניים” בלי לחשוב קדימה.

---

## שינוי החשיבה

כאן משהו בראש השתנה,  
לא שאלתי יותר, “איך אני מוסיף פיצ׳ר?”  
אלא, “באיזו שכבה הפיצ׳ר הזה חי?”

וזה בדיוק ההבדל בין מפתח רגיל לארכיטקט.

בנקודה הזו כבר לא היה ספק, השלד של המערכת קיים.  
אבל עדיין אין קוד אמיתי שמחבר את הכל.
